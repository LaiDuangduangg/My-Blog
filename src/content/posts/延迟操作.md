## Description

给定一个长度为 $N$ 的整数数列：$A_1, A_2, \dots, A_N$。你要重复以下操作 $K$ 次：

每次选择数列中最小的整数（如果最小值不止一个，选择最靠前的），将其删除。并把与它相邻的整数加上被删除的数值。

输出 $K$ 次操作后的序列。

## Input

第一行包含两个整数 $N$ 和 $K$。

第二行包含 $N$ 个整数，$A_1, A_2, A_3, \dots, A_N$。

## Output

输出 $N - K$ 个整数，中间用一个空格隔开，代表 $K$ 次操作后的序列。

## Sample 1

**Input**
```text
5 3
1 4 2 8 7
```

**Input**
```text
17 7
```


## Hint

**【样例说明】**

数列变化如下，中括号里的数是当此操作中被选择的数：

```text
1   [1] 4 2 8 7
2   5 [2] 8 7
3   [7] 10 7
4   17 7
```
**【评测用例规模与约定】**

对于 $20\%$ 的数据，$1 \le K < N \le 10^4$。

对于 $100\%$ 的数据，$1 \le K < N \le 5 \times 10^5$，$0 \le A_i \le 10^8$。

### 题目分析：

* 如果是 $10^4$ 的规模，允许算法的最坏时间复杂度达到 $O(N^2)$ 左右。但对于 $5 \times 10^5$ 的规模，哪怕是 $O(N \sqrt{N})$ 都大概率会 TLE（超时），我们必须把时间复杂度死死压在 $O(N \log N)$ 或者更低。

* 既然题目让我找最小值并合并，那我就用最基础的数据结构 $std::vector$。
每次遍历整个` vector `找到最小值，记录它的下标。然后把它的值加给 `[index-1]` 和 `[index+1]`，最后调用` vector.erase(index)`把这个数删掉。

## 复杂度分析：

1. **找最小值简直非常慢**：每次遍历找最小值需要 $O(N)$, 即便我们可以用优先队列找到这个最小值，但是对应到原来那个数组的位置是非常难的。

2. **删除元素及其慢**：我们要找到删除这个数的左邻居和右邻居，找到之后还要修改他们的值，把他们从优先队列里面拉出来，然后改完再丢回去，等到黄花菜都凉了.


暴力的痛点在于"找最小值慢"。提到快速找最小值，大家都能想到是**优先队列（小根堆）**。

把所有数字塞进 `priority_queue`，每次 `pop` 出来的就是最小值，时间复杂度瞬间降维到 $O(\log N)$！


当你弹出最小值准备合并时，致命问题出现了：**合并操作会改变邻居的值**。

假设原数组是 `[1, 4, 2, 8]`。

1. 优先队列弹出 `1`，你需要在原数组里把 `1` 加给右邻居 `4`，`4` 变成了 `5`。

2. **死局**：此时优先队列里还存着那个旧的 `4`。你根本无法在 $O(\log N)$ 的时间里，去优先队列内部找到那个 `4` 并把它改成 `5`！因为 `std::priority_queue` 不支持内部元素的查找和修改。

到这里，一般选手的思路就会断掉。




## 先解决邻居快速查找：

使用双向链表。但在算法竞赛中，直接 `new Node` 写指针非常容易出错且常数大。最优雅的写法是**数组模拟链表**：

开两个数组 `L[i]` 和 `R[i]`，分别表示第 $i$ 个位置的左邻居和右邻居的真实索引。

*   **初始化**：`L[i] = i - 1`，`R[i] = i + 1`。
*   **删除第 $i$ 个节点时**：只需要让它的左右邻居互相牵手。
    *   `R[L[i]] = R[i]` （我左邻居的右边，变成了我的右边）
    *   `L[R[i]] = L[i]` （我右邻居的左边，变成了我的左边）

这样，无论中间删除了多少个数，找邻居永远是 $O(1)$。

## 再解决一下旧数据怎么处理？
**这才是最难的部分**


现在回到让我们卡死的那个优先队列问题：**那堆里的旧数据改不掉怎么办？**

**延迟操作的精髓就在这里要体现了：** 如果你无法修改它，那就抛弃它、无视它。

我们设立一个“实时更新的数组”：

1. 开一个真实的数值数组 `V[i]`，记录第 $i$ 个位置当前最准确的数值。
2. 开一个状态数组 `stale[i]`，记录第 $i$ 个位置是否已经被删除了。

* 因为不管是邻居值的修改，还是当前值的删除，我们都是没有办法时时刻刻保证优先队列里面的数据是干净的，所以
当我们发现邻居的值变了（比如第 `idx` 个位置的值增加了 `val`），我们**不去管优先队列里那个旧的记录**。

我们直接把新值存入真理数组 `V[idx] += val`，然后大胆地把这个新的状态 `{V[idx], idx}` 再次 `push` 进优先队列。
我们只需要保证优先队列里面有当前邻居的真实值就好了，然后我们用一些手段排除掉旧的数据，那这跟原来直接删除没什么区别了。

## 那队列里乱七八糟的旧数据什么时候处理？
等它们自己“出头”的时候处理。
每次从优先队列 `top` 弹出一个元素 `{cur_val, cur_id}` 时，我们立刻进行“数据检验”：

*   **校验 1：** 查 `stale[cur_id]`。如果它是 `true`，说明这个位置早就被删除了。这是个死人，扔掉。
*   **校验 2：** 比对 `cur_val` 和 `V[cur_id]`。如果不相等，说明这是一条过期的历史记录，它早就被后来的更新替代了，扔掉。

只有当这两个校验都通过时，才能证明，当前这个数据是符合真实情况的，不然你都没办法保证你当前这个数据就是最小的

## 为了让我们解释的更加清楚，我决定来分析一下这个**校验 1：**和  **校验 2：**的情况
**校验 2：** 
**判断条件：** `cur_val != V[cur_id]`

**作用：** 拦截那些"活着，但数据已经过期"的节点。

### 什么时候会出现？

假设数组里有个元素：2号位，当前值是 5。

初始状态下，优先队列里存了一个它的初始记录：`{val: 5, id: 2}`。

1. 突然，2号位的左邻居（1号位）被选中删除了，1号位把自己的值 3 加给了2号位。
2. 此时，真实数组里 `V[2]` 变成了 $5 + 3 = 8$。
3. 按照我们的算法，我们会把更新后的状态 `{val: 8, id: 2}` 压入优先队列。

### 注意此时优先队列里的奇观：

队列里同时存在了 **两个** 2号位的数据：

*   旧记录（历史残影）：`{val: 5, id: 2}`
*   新记录（真实状态）：`{val: 8, id: 2}`

因为是小根堆，值小的排在前面，所以那个旧的 `{val: 5, id: 2}` 会**先被弹出来**！

当它弹出来时，程序一看：你告诉我你叫2号位，值是 5。但是我去真实数组 `V[2]` 里一查，人家明明已经是 8 了！

这就说明，弹出来的这个 5 是一张**过时的数据，是一份假的数据**。

**触发校验 2，直接扔掉**

## 再说【校验 1：死亡证明拦截】

**判断条件：** `stale[cur_id] == true`

**作用：** 拦截那些“已经彻底死亡（被删除过）”的节点。

你可能会问：既然有校验 2 把关，值不对就会被扔掉，那一个死掉的节点，它的值肯定也是旧的呀，校验 2 不就顺手把它拦住了吗？为什么还要专门搞个 `stale` 数组（校验 1）？

**什么时候校验 2 会失效，必须靠校验 1 来救命？**

来看一个非常极端的诡异情况：题目约定数值 $A_i$ 可以是 0！

假设 3 号位，当前值是 10。

1. 它的左邻居被删了，左邻居的值恰好是 0。
2. 3 号位加上了 0，真实数组 `V[3]` 依然是 10。
3. 按照算法逻辑，我们会把 `{val: 10, id: 3}` 再次压入队列。

此时队列里有两个一模一样的记录：

*   初始记录：`{val: 10, id: 3}`
*   加上 0 后的新记录：`{val: 10, id: 3}`

接下来发生的事情极其危险：

1. 第一次弹出 `{val: 10, id: 3}`。程序去查 `V[3]`，发现也是 10。校验 2 通过！
2. 它是合法的最小值，我们把它删了，合并给它的邻居。并在 `stale[3]` 上盖个章：**死亡**。
3. 过了一会，队列里**第二个** `{val: 10, id: 3}` 浮上来了！被弹出了！
4. 程序去查验校验 2：弹出的 `cur_val` 是 10，真实数组 `V[3]` 停留在它死前的那一刻，依然是 10！**校验 2 完美通过（被骗过去了）！**

如果没有校验 1，程序就会把 3 号位再删除一次，再合并一次，不仅凭空多出数值，还会把双向链表的指针彻底搞乱，导致程序死循环或段错误。

**这就是校验 1 的终极意义：**

## 但是这操作说的貌似让人有点懵？
1. 所以我们会发现，state也只是对真实数组的操作而已，我们只需要保证真实数据那组是实时更新的就好对吧。
2. 因此我们可以在删掉一个数的时候，把原先数组对应的值设置成0就好了，这样真实数组就可以保证真实，(原先的操作只是说我删掉之后没有给他置零，所以用一个state数组来标记他已经被删了)
3. 所以说，state或者置零操作我们二选一，就可以保证真实数组 `V[]`的确是真实的

## 核心代码部分：
```cpp
#include <bits/stdc++.h>
using namespace std;

const int MAXN = 5e5 + 10;
int L[MAXN];
int R[MAXN]; // 双向链表
long long V[MAXN];
vector <bool> state (MAXN, true);

struct infoma{
    int index;
    long long val;

    // <表示下沉
    bool operator <(const infoma& other) const{
        if (val != other.val) {
            return val > other.val;//  表示大的下沉
        }
        return index > other.index; // 下标大的下沉
    }
};

int main ()
{
    ios::sync_with_stdio (false);
    cin.tie (0);

    int n, k;
    cin >> n >> k;

    vector <int> a(n + 1);
    for (int i = 1; i <= n; i ++) {
        cin >> a[i];
    }
    for (int i = 1; i <= n; i ++) {
        L[i] = i - 1;
        R[i] = i + 1;
    }
    L[1] = 0;
    R[n] = 0;

    priority_queue <infoma> pq;
    for (int i = 1; i <= n; i ++) {
        pq.push ({i, a[i]});
        V[i] = a[i];
    }
    for (int i = 0; i < k; i ++) {
        int sur_index;
        long long sur_val;
        while (!pq.empty ()) {
            auto [cur_index, cur_val] = pq.top ();
            pq.pop ();

            if (!state[cur_index]) continue;
            if (V[cur_index] != cur_val) continue;

            sur_index = cur_index;
            sur_val = cur_val;
            break;
        }
        state[sur_index] = false; // 删除掉
        
        int L_nei = L[sur_index];
        int R_nei = R[sur_index];

        if (L_nei != 0) {// 该点左邻居存在
            V[L_nei] += sur_val;
            R[L_nei] = R_nei;
            pq.push ({L_nei, V[L_nei]});
        }

        if (R_nei != 0) {// 该点右邻居存在
            V[R_nei] += sur_val;
            L[R_nei] = L_nei;
            pq.push ({R_nei, V[R_nei]});    
        }
    }

    for (int i = 1; i <= n; i ++) {
        if (state[i]) {
            cout << V[i] << " ";
        }
    }
    cout << '\n';
    return 0;
} 
```
