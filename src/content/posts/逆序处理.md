# 题目描述

有一个 $N$ 行 $N$ 列的方格棋盘。

起初，棋盘上一个棋子都没有。

从这个状态开始，高桥依次对棋盘执行 $M$ 次操作。第 $i$ 次操作（$1 \le i \le M$）如下：

- 先把第 $R_i$ 行（从上往下数）上所有的棋子全部拿掉。
- 接着把第 $C_i$ 列（从左往右数）上所有的棋子全部拿掉。
- 最后，在第 $R_i$ 行第 $C_i$ 列的位置放一个棋子。

请你输出执行完 $M$ 次操作后，棋盘上剩下的棋子数。

## 约束条件

- $1 \le N \le 3 \times 10^5$
- $1 \le M \le 3 \times 10^5$
- $1 \le R_i \le N$
- $1 \le C_i \le N$
- 所有输入均为整数。

## 输入格式

输入从标准输入中读取，格式如下：

```text
N M
R_1 C_1
R_2 C_2
.....
.....
R_M C_M
```

## 输出格式

输出执行完 $M$ 次操作后，棋盘上剩余的棋子数。

## 样例 1

### 输入

```text
3 6
1 1
1 2
3 3
3 2
1 3
1 3
```

### 输出

```text
2
```

最开始，棋盘是一个 $3$ 行 $3$ 列的空棋盘。每次操作移除和放置棋子的情况如下。

下面用第 $i$ 行第 $j$ 列的格子记作格子 $(i,j)$。

- 第 $1$ 次操作，在格子 $(1,1)$ 放了一个棋子。
- 第 $2$ 次操作，先把格子 $(1,1)$ 的棋子拿掉，再在格子 $(1,2)$ 放一个棋子。
- 第 $3$ 次操作，在格子 $(3,3)$ 放了一个棋子。
- 第 $4$ 次操作，先把格子 $(1,2)$ 和格子 $(3,3)$ 的棋子都拿掉，再在格子 $(3,2)$ 放一个棋子。
- 第 $5$ 次操作，在格子 $(1,3)$ 放了一个棋子。
- 第 $6$ 次操作，先把格子 $(1,3)$ 的棋子拿掉，再在格子 $(1,3)$ 重新放一个棋子。

最后，格子 $(1,3)$ 和格子 $(3,2)$ 中各有一个棋子，所以输出 $2$。

# 逆序处理：棋盘上的行列删除操作

## 一、核心考点

一句话概括：

> 不要正向维护棋盘上的所有棋子，而要倒序判断每次操作放下的棋子能否存活到最后。

这道题的关键是将关注点从“每次操作删除了哪些棋子”，转移到：

> 当前操作放下的这个棋子，之后会不会被其他操作删除？

---

## 二、最直接的暴力想法

最直接的做法是开一个二维数组模拟整个棋盘。

对于每次操作 $(R_i,C_i)$：

1. 枚举第 $R_i$ 行，将这一行的棋子全部删除；
2. 枚举第 $C_i$ 列，将这一列的棋子全部删除；
3. 在 $(R_i,C_i)$ 放置一个棋子。

每次操作都需要枚举一整行和一整列，时间复杂度为：

```text
O(N)
```

总时间复杂度为：

```text
O(NM)
```

由于：

```text
N,M <= 3×10^5
```

显然无法通过。

而且如果直接开一个 $N\times N$ 的棋盘，空间复杂度也会达到：

```text
O(N^2)
```

同样无法接受。

---

## 三、如果使用行桶和列桶

看到题目中的操作都是清空某一行或某一列，一个比较自然的想法是：

- 用一个桶记录每一行有哪些棋子；
- 再用一个桶记录每一列有哪些棋子。

但是，一个棋子同时属于一行和一列。

假如我们清空某一行，那么不仅要从“行桶”中删除这些棋子，还必须找到这些棋子分别属于哪些列，并同步修改对应的“列桶”。

同理，清空某一列时，也必须同步修改对应的所有行。

也就是说：

```text
删除一行会影响很多列；
删除一列会影响很多行。
```

行和列之间存在紧密的依赖关系。

如果没有额外的数据结构支持，一次删除操作仍然可能涉及大量棋子，最坏复杂度依然无法接受。

因此，正向维护“棋盘上现在有哪些棋子”并不是一个合适的方向。

---

## 如果我们把目光放在每个棋子身上

第 $i$ 次操作会在：

```text
(R_i,C_i)
```

放下一个棋子。

考虑这个棋子最后能否留在棋盘上。

这个棋子会被删除，当且仅当在第 $i$ 次操作之后，出现了下面至少一种情况：

1. 某次操作选择了第 $R_i$ 行；
2. 某次操作选择了第 $C_i$ 列。

因为后面的操作只要使用同一行或同一列，就会将 $(R_i,C_i)$ 上的棋子删除。

所以，第 $i$ 次操作放下的棋子能够存活到最后，当且仅当：

```text
后面没有操作使用第 R_i 行，
并且后面没有操作使用第 C_i 列。
```

也可以写成：

```text
不存在 j > i，使得 R_j = R_i 或 C_j = C_i。
```

因此，一个棋子最终是否存在，只依赖于它后面的操作。


---

## 正序处理

如果从前往后处理第 $i$ 次操作，为了判断当前棋子能否存活，我们需要提前知道：

```text
后面的所有操作使用了哪些行和列。
```

可以先统计所有操作中每一行和每一列出现的次数，然后随着当前位置向后移动，不断将已经经过的操作对应计数减一。

但是这种写法需要维护“剩余操作中某行、某列出现了多少次”，不能只使用简单的布尔标记。

而且处理顺序与我们需要的信息方向相反：

```text
从前往后处理，
却需要知道当前位置后面的信息。
```

虽然这种方法也能实现，但不够自然。

如果把操作顺序倒过来，情况就会简单很多。

---

## 倒序处理

我们从第 $M$ 次操作开始，一直处理到第 $1$ 次操作。

定义：

```cpp
userow[r]
```

表示在当前操作之后，是否已经有操作使用过第 $r$ 行。

定义：

```cpp
usecol[c]
```

表示在当前操作之后，是否已经有操作使用过第 $c$ 列。

由于我们是倒序处理，所以“已经处理过的操作”恰好就是原顺序中位于当前操作之后的操作。

对于当前操作 $(x,y)$：

### 情况一：这一行和这一列都没有被后面的操作使用

即：

```cpp
userow[x] == false && usecol[y] == false
```

说明后面没有任何操作会清空第 $x$ 行或第 $y$ 列。

因此，当前操作放在 $(x,y)$ 的棋子能够一直保留到最后，答案加一。

### 情况二：这一行或这一列已经被后面的操作使用

即：

```cpp
userow[x] == true || usecol[y] == true
```

说明后面存在某次操作会清空它所在的行或列。

因此，当前棋子最终会被删除，不能计入答案。

---


## 代码如下:
```cpp
#include <bits/stdc++.h>
using namespace std;

constexpr int MAXN = 3e5 + 10;
bool userow[MAXN]; // 被清空过的行
bool usecol[MAXN]; // 被清空过的列

int main ()
{
    ios::sync_with_stdio (false);
    cin.tie (0);

    int n, m;
    cin >> n >> m;

    int ans = 0;
    stack <pair <int, int>> st;
    for (int i = 0; i < m; i ++) {
        int x, y;
        cin >> x >> y;
        st.push ({x, y});
    }
    while (!st.empty ()) {
        auto [x, y] = st.top ();
        st.pop ();

        if (userow[x] == 0 && usecol[y] == 0) {
            ans ++;
        }
        userow[x] = true;
        usecol[y] = true;
    }
    cout << ans << '\n';
    return 0;
}
```

---
## 复杂度分析

每个操作只会：

1. 入栈一次；
2. 出栈一次；
3. 进行常数次判断和标记。

因此时间复杂度为：

```text
O(M)
```
